# hitl

## Overview
`hitl-mcp` lets agents ask humans structured question groups while keeping identifier ownership and pending-state tracking on the server.

The current model is explicit:

1. Create a pending question group with `hitl_create_question_group`.
2. Keep the returned server-generated `question_group_id`.
3. Wait only when blocking semantics are actually needed with `hitl_wait_question_group`.
4. Recover current pending state for the caller scope with `hitl_get_current_question_group`.

Use this skill for every human-facing question, including one-off text questions, so answers are tracked through stable server-owned IDs and per-session pending state.

## When to Use
- Any time the agent needs input from a human, even a single quick question
- When you need structured question groups for UI rendering or downstream processing
- When you need the server to own `question_group_id`
- When human input must be scoped to the current authenticated agent identity and agent session
- When the agent may need either immediate return or an explicit later wait point

## When Not to Use
- Purely internal reasoning with no human input
- Fully automated flows that never require human confirmation

## Core Model
- `agent_identity` is derived from MCP connection authentication.
- `agent_session_id` is derived from the configured connection header, default `x-agent-session-id`.
- `question_group_id` is always generated by the server.
- At most one `pending` question group may exist for a given `(agent_identity, agent_session_id)` pair.

This means the agent should never invent the group ID itself and should treat caller scope as transport-derived, not tool-input-derived.

## Core Workflow (Agent Side)
1. Call `hitl_create_question_group` with a **question group** payload and no caller-generated `question_group_id`.
2. Store the returned server-generated `question_group_id`.
3. If the flow needs to continue asynchronously, keep going and let the UI/backend work against that group ID.
4. If the flow needs blocking semantics, call `hitl_wait_question_group` with that `question_group_id`, or omit the ID to wait on the current pending group for the caller scope.
5. Handle the terminal status: `answered`, `cancelled`, or `expired`.

## Key Rules
- Always use a question group, even for a single human question.
- Never call the old `hitl_ask_question_group`; it is obsolete in this project.
- Never generate `question_group_id` on the agent side.
- Keep `question_id` stable and unique within the question group you send.
- Treat create and wait as separate actions; create does not block.
- Treat terminal states as final; do not expect a group to revert to `pending`.
- Remember that the current caller scope may already own a pending group; avoid accidentally creating a second one.

## Tool Quick Reference
- `hitl_create_question_group`: create a `pending` question group and return immediately. Required: `title`, `questions`.
- `hitl_wait_question_group`: wait for a specific `question_group_id`, or the caller's current pending group if no ID is provided.
- `hitl_get_current_question_group`: fetch the current pending group for the caller scope.
- `hitl_get_question_group_status`: check group status by `question_group_id`.
- `hitl_get_question`: fetch a question definition by `question_id`.
- `hitl_cancel_question_group`: cancel a pending group by `question_group_id` (optional `reason`).

Full schema and constraints: `docs/api/mcp-tools.md`.

## Minimal Example

**Create a single-question group:**

```json
{
  "title": "Release Decision",
  "questions": [
    {
      "question_id": "q_canary",
      "type": "single_choice",
      "title": "Can we start canary deployment?",
      "options": [
        { "value": "yes", "label": "Yes" },
        { "value": "no", "label": "No" }
      ]
    }
  ]
}
```

**Create response (example):**

```json
{
  "question_group_id": "qg_01JXYZ...",
  "agent_identity": "api_key:agent-key",
  "agent_session_id": "session-123",
  "status": "pending"
}
```

**Wait for the current pending group:**

```json
{}
```

**Terminal response (example):**

```json
{
  "question_group_id": "qg_01JXYZ...",
  "status": "answered",
  "answers": {
    "q_canary": { "value": "yes" }
  }
}
```

## Common Mistakes
- Calling the removed `hitl_ask_question_group`
- Generating `question_group_id` on the agent side
- Forgetting that create and wait are separate operations
- Forgetting that caller scope comes from authentication plus session header, not tool input
- Creating another group when the same caller scope already has a pending one
- Treating `cancelled` or `expired` as recoverable
