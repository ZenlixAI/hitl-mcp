# hitl

## Overview
`hitl-mcp` lets agents ask humans structured requests while keeping identifier ownership and pending-state tracking on the server.

The current model is explicit:

1. Create a pending request with `hitl_create_request`.
2. Keep the returned server-generated `request_id`.
3. Wait only when blocking semantics are actually needed with `hitl_wait_request`.
4. Recover current pending state for the caller scope with `hitl_get_current_request`.

Use this skill for every human-facing question, including one-off text questions, so answers are tracked through stable server-owned IDs and per-session pending state.

## When to Use
- Any time the agent needs input from a human, even a single quick question
- When you need structured requests for UI rendering or downstream processing
- When you need the server to own `request_id`
- When human input must be scoped to the current authenticated agent identity and agent session
- When the agent may need either immediate return or an explicit later wait point

## When Not to Use
- Purely internal reasoning with no human input
- Fully automated flows that never require human confirmation

## Core Model
- `agent_identity` is derived from MCP connection authentication.
- `agent_session_id` is derived from the configured connection header, default `x-agent-session-id`.
- `request_id` is always generated by the server.
- At most one `pending` request may exist for a given `(agent_identity, agent_session_id)` pair.

This means the agent should never invent the request ID itself and should treat caller scope as transport-derived, not tool-input-derived.

## Core Workflow (Agent Side)
1. Call `hitl_create_request` with a **request** payload and no caller-generated `request_id`.
2. Store the returned server-generated `request_id`.
3. If the flow needs to continue asynchronously, keep going and let the UI/backend work against that request ID.
4. If the flow needs blocking semantics, call `hitl_wait_request` with that `request_id`, or omit the ID to wait on the current pending request for the caller scope.
5. Handle the terminal status: `answered`, `cancelled`, or `expired`.

## Key Rules
- Always use a request, even for a single human question.
- Never generate `request_id` on the agent side.
- Keep `question_id` stable and unique within the request you send.
- For optional questions (`required: false`), design UX with an explicit "skip/ignore" action.
- Treat create and wait as separate actions; create does not block.
- Treat terminal states as final; do not expect a request to revert to `pending`.
- Remember that the current caller scope may already own a pending request; avoid accidentally creating a second one.

## Tool Quick Reference
- `hitl_create_request`: create a `pending` request and return immediately. Required: `title`, `questions`.
- `hitl_wait_request`: wait for a specific `request_id`, or the caller's current pending request if no ID is provided.
- `hitl_get_current_request`: fetch the current pending request for the caller scope.
- `hitl_get_request_status`: check request status by `request_id`.
- `hitl_get_question`: fetch a question definition by `question_id`.
- `hitl_cancel_request`: cancel a pending request by `request_id` (optional `reason`).

Full schema and constraints: `docs/api/mcp-tools.md`.

## Minimal Example

**Create a single-request:**

```json
{
  "title": "Release Decision",
  "questions": [
    {
      "question_id": "q_canary",
      "type": "single_choice",
      "title": "Can we start canary deployment?",
      "options": [
        { "value": "yes", "label": "Yes" },
        { "value": "no", "label": "No" }
      ]
    }
  ]
}
```

**Create response (example):**

```json
{
  "request_id": "qg_01JXYZ...",
  "agent_identity": "api_key:agent-key",
  "agent_session_id": "session-123",
  "status": "pending"
}
```

**Wait for the current pending request:**

```json
{}
```

**Terminal response (example):**

```json
{
  "request_id": "qg_01JXYZ...",
  "status": "answered",
  "answers": {
    "q_canary": { "value": "yes" }
  },
  "skipped_question_ids": []
}
```

## Optional Question Skip Rule
- The finalize API now enforces explicit handling for optional questions.
- Optional questions must be either:
  - answered in `answers`, or
  - explicitly listed in `skipped_question_ids`.
- Required questions cannot be listed in `skipped_question_ids`.

If your human UI allows "leave blank", map that action to an explicit skip and include that `question_id` in `skipped_question_ids` during finalize.

## Common Mistakes
- Generating `request_id` on the agent side
- Forgetting that create and wait are separate operations
- Forgetting that caller scope comes from authentication plus session header, not tool input
- Creating another request when the same caller scope already has a pending one
- Treating `cancelled` or `expired` as recoverable
- Sending finalize payload without `skipped_question_ids` when optional questions are unanswered
