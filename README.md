## Why This Exists

In complex agent workflows, plain assistant text questions are not enough:

1. They are not strongly structured for UI rendering.
2. They do not provide stable lifecycle tracking for question groups and questions.
3. They do not cleanly support the flow of user answer -> backend post-processing -> final server confirmation.

Older blocking-only HITL flows also create integration friction for Agent platforms:

1. The Agent ends up owning identifiers it should not trust.
2. Session scoping becomes ambiguous across reconnects and retries.
3. The client cannot reliably observe a `pending` question group before the final human answer arrives.

`hitl-mcp` solves this with a server-owned identity and lifecycle model:

1. `agent_identity` is derived from MCP connection authentication.
2. `agent_session_id` is derived from a stable connection header, default `x-agent-session-id`.
3. `question_group_id` is always generated by the server.
4. One `pending` question group may exist for a given `(agent_identity, agent_session_id)` pair.

## Product Goals

- Always ask in **Question Group** form, never single loose questions.
- Keep question-group ownership on the server.
- Let Agent platforms observe a `pending` question group before the human answer is finalized.
- Support rich question types:
  - `single_choice`
  - `multi_choice`
  - `text`
  - `boolean`
  - `range`
- Support required (default) and optional questions.
- Preserve structured metadata (`tags`, `extra`) for group and question definitions.
- Provide an ID-oriented HTTP control plane.
- Support persistence and TTL through KV storage (Redis).

## Who Uses It

### For Agent Developers

You register `hitl-mcp` in your MCP client and call tools like:

- `hitl_create_question_group`
- `hitl_wait_question_group`
- `hitl_get_current_question_group`
- `hitl_get_question_group_status`
- `hitl_get_question`
- `hitl_cancel_question_group`

Your Agent does not need to invent its own question-group IDs or custom pending-state protocol.

### For Backend Developers

You integrate HTTP endpoints to finalize answers after your own post-processing:

- `PUT /api/v1/question-groups/{id}/answers/finalize`
- `POST /api/v1/question-groups/{id}/cancel`
- `POST /api/v1/question-groups/{id}/expire`
- `GET /api/v1/question-groups/current`
- `GET /api/v1/question-groups/{id}`
- `GET /api/v1/questions/{id}`

No list APIs are exposed by design.

## User Story (End-to-End)

1. Agent calls `hitl_create_question_group` with structured questions.
2. The server authenticates the caller, reads `x-agent-session-id`, generates `question_group_id`, and stores the group as `pending`.
3. Client or backend receives the returned `question_group_id` immediately and can render or track the pending state.
4. When blocking semantics are needed, Agent calls `hitl_wait_question_group`.
5. User answers in the client UI.
6. Your backend post-processes the raw user response.
7. Backend calls `answers/finalize` HTTP API.
8. `hitl-mcp` validates answers:
   - invalid -> returns `422 ANSWER_VALIDATION_FAILED`, keeps `pending`
   - valid -> transitions to `answered`, wakes blocked MCP wait call
9. Agent resumes with finalized answer payload.

## Key Design Decisions

### 1) Server-Generated Question Group IDs

`question_group_id` is never supplied by the Agent. This keeps object ownership and lifecycle authority on the server.

### 2) Stable Caller Scope

Caller scope is defined by trusted transport context:

- `agent_identity` from MCP connection authentication
- `agent_session_id` from a stable connection header

This avoids relying on tool-input metadata for session identity.

### 3) Split Create And Wait

The MCP surface is intentionally explicit:

- `create` creates a `pending` group and returns immediately
- `wait` blocks only when the Agent actually wants blocking semantics
- `get_current` lets the Agent recover current pending state for its caller scope

### 4) One Pending Group Per Caller Scope

At most one `pending` question group may exist for a given `(agent_identity, agent_session_id)` pair.

This keeps client-side pending state deterministic.

### 5) Validation + Idempotency

Finalize validates required fields and type/range constraints. Duplicate create/finalize requests can be idempotent via `idempotency_key`.

### 6) Storage Strategy

- In-memory repository for local/dev flow.
- Redis repository for persistent KV state and TTL.
- Pending-scope and idempotency indexes for caller-scoped lookup.

## Current Capabilities

- Structured question schemas and validators.
- MCP tools for create/wait/current/status/question/cancel.
- HTTP routes for query/current/finalize/cancel/expire/healthz/readyz/metrics.
- State machine: `pending -> answered|cancelled|expired`.
- Wait/notify mechanism for explicit MCP waits.
- API key middleware (`x-api-key`) for protected HTTP APIs (when enabled).
- Request-context extraction for authenticated caller identity and session header.
- Redis-backed repository with tests.

## Configuration

Environment variables:

- `PORT` (default `3000`)
- `MCP_URL` (default `http://localhost:3000`)
- `HITL_PENDING_MAX_WAIT_SECONDS` (`0` means infinite wait)
- `HITL_STORAGE` (`memory` | `redis`)
- `HITL_REDIS_URL` (default `redis://127.0.0.1:6379`)
- `HITL_REDIS_PREFIX` (default `hitl`)
- `HITL_TTL_SECONDS` (default `604800`)
- `HITL_API_KEY` (when set, protected HTTP routes require `x-api-key`)
- `HITL_AGENT_AUTH_MODE` (`api_key` | `bearer`)
- `HITL_AGENT_SESSION_HEADER` (default `x-agent-session-id`)
- `HITL_CREATE_CONFLICT_POLICY` (`error` | `reuse_pending`)
- `HITL_SERVER_NAME` (optional, server name override)
- `HITL_SERVER_VERSION` (optional, server version override)
- `HITL_HTTP_HOST` (optional, HTTP bind host)
- `HITL_HTTP_API_PREFIX` (optional, default `/api/v1`)
- `HITL_ANSWERED_RETENTION_SECONDS` (optional)
- `HITL_LOG_LEVEL` (`debug|info|warn|error`)
- `HITL_ENABLE_METRICS` (`true|false`)

Configuration precedence:

- `env` > `.env` > `config/hitl-mcp.yaml` > defaults

Tool: `hitl_create_question_group`

```json
  "title": "Release Decision",
  "questions": [
    {
      "question_id": "q_canary",
      "type": "single_choice",
      "title": "Can we start canary deployment?",
      "options": [
        { "value": "yes", "label": "Yes" },
        { "value": "no", "label": "No" }
      ]
    }
  ]
}
```

### Agent: Wait for Current Pending Group

Tool: `hitl_wait_question_group`

```json
{}
```

### Backend: Finalize Answer

```bash
curl -X PUT "http://localhost:3000/api/v1/question-groups/<question_group_id>/answers/finalize" \
  -H "Content-Type: application/json" \
  -H "x-api-key: ${HITL_API_KEY}" \
  -d '{
    "idempotency_key": "idem-release-001",
    "answers": {
      "q_canary": { "value": "yes" }
    },
    "finalized_by": "release-orchestrator"
  }'
```

## MCP Tools

### `hitl_create_question_group`

Purpose:

- Create a `pending` question group for the current authenticated caller scope and return immediately.

Input (important fields):

- `title` (string, required)
- `description` (string, optional, markdown)
- `tags` (string[], optional)
- `extra` (object, optional)
- `ttl_seconds` (number, optional)
- `questions` (array, required)
- `idempotency_key` (string, optional)

Caller scope:

- `agent_identity` is derived from authentication
- `agent_session_id` is derived from the configured session header

Output:

- Server-generated `question_group_id`
- `status: "pending"`
- caller scope fields
- timestamps and question payload

### `hitl_wait_question_group`

Purpose:

- Wait for a question group to reach a terminal state (`answered`, `cancelled`, `expired`).

Input:

- `question_group_id` (string, optional)

Behavior:

- If `question_group_id` is provided, wait for that group.
- If omitted, wait for the caller's current pending group.

Output:

- On finalize success: status `answered` + validated `answers`
- On cancel: status `cancelled`
- On expire: status `expired`

Notes:

- This tool is intentionally blocking.
- If `HITL_PENDING_MAX_WAIT_SECONDS=0`, wait is unbounded.

### `hitl_get_current_question_group`

Purpose:

- Return the current pending question group for the caller scope.

Output:

- Full pending group object when present.
- `PENDING_GROUP_NOT_FOUND` when no pending group exists for the caller scope.

### `hitl_get_question_group_status`

Purpose:

- Query group lifecycle state by ID.

Input:

- `question_group_id` (string)

Output:

- `question_group_id`
- `status` (`pending|answered|cancelled|expired`)
- `updated_at`

### `hitl_get_question`

Purpose:

- Fetch one question definition by `question_id`.

Input:

- `question_id` (string)

Output:

- Full question object.

### `hitl_cancel_question_group`

Purpose:

- Cancel a pending group and wake blocked wait calls.

Input:

- `question_group_id` (string)
- `reason` (string, optional)

Output:

- `status: "cancelled"`
- Optional `reason`

## HTTP API

Base path:

- `/api/v1`

Common response envelope:

```json
{
  "request_id": "uuid",
  "success": true,
  "data": {},
  "error": null
}
```

### `GET /healthz`

Purpose:

- Liveness check.

Success:

- `200` with `data.status = "ok"`.

### `GET /readyz`

Purpose:

- Readiness probe for deployment systems.

Behavior:

- `200` with `status=ready` when backend storage is usable.
- `503` with `status=not_ready` when backend is not ready.

### `GET /metrics`

Purpose:

- Operational metrics endpoint.

Response includes:

- `counters.finalize_validation_failed_total`
- `counters.finalize_success_total`
- `gauges.pending_count`
- `histograms.wait_duration_ms` (count/min/max/avg)

### `GET /question-groups/current`

Purpose:

- Fetch the current pending question group for the authenticated caller scope.

Headers:

- `x-api-key` when API key auth is enabled
- configured session header, default `x-agent-session-id`

Errors:

- `401 UNAUTHORIZED`
- `400 AGENT_SESSION_ID_REQUIRED`
- `404 PENDING_GROUP_NOT_FOUND`

### `GET /question-groups/{question_group_id}`

Purpose:

- Fetch a question group by ID.

Errors:

- `404 QUESTION_GROUP_NOT_FOUND`

### `GET /questions/{question_id}`

Purpose:

- Fetch one question by ID.

Errors:

- `404 QUESTION_NOT_FOUND`

### `PUT /question-groups/{question_group_id}/answers/finalize`

Purpose:

- Submit final, post-processed answers from backend.

Request body:

```json
{
  "idempotency_key": "idem-1",
  "answers": {
    "q_1": { "value": "A" }
  },
  "finalized_by": "agent-server",
  "extra": {}
}
```

Behavior:

- Validates answer type/range/required rules.
- If invalid: keeps group `pending`.
- If valid: transitions group to `answered` and wakes blocked MCP wait calls.

Success:

- `200` with `status: "answered"` and `answered_question_ids`.

Validation error:

- `422 ANSWER_VALIDATION_FAILED` with per-question details.

Idempotency:

- Reusing same `idempotency_key` returns the same finalize result.

### `POST /question-groups/{question_group_id}/cancel`

Purpose:

- Cancel a pending group.

Behavior:

- Marks group `cancelled`.
- Wakes blocked MCP wait calls with terminal state.

### `POST /question-groups/{question_group_id}/expire`

Purpose:

- Force-expire a group.

Behavior:

- Marks group `expired`.
- Wakes blocked MCP wait calls with terminal state.

### Authentication

When `HITL_API_KEY` is set:

- Protected routes require `x-api-key`.
- Missing/invalid key returns `401 UNAUTHORIZED`.

### Error Codes

- `QUESTION_GROUP_NOT_FOUND` (`404`)
- `QUESTION_NOT_FOUND` (`404`)
- `PENDING_GROUP_NOT_FOUND` (`404`)
- `AGENT_IDENTITY_REQUIRED` (`401`)
- `AGENT_SESSION_ID_REQUIRED` (`400`)
- `ANSWER_VALIDATION_FAILED` (`422`)
- `UNAUTHORIZED` (`401`)
