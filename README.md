# hitl-mcp

A Human-in-the-loop MCP server for agentic systems.

`hitl-mcp` lets an Agent create structured question groups, wait on them, and resume only after a server-side finalized human answer is submitted through HTTP.

## Core Model

- `agent_identity` is derived from MCP connection authentication.
- `agent_session_id` is derived from the configured connection header, default `x-agent-session-id`.
- `question_group_id` is always generated by the server.
- One `pending` question group may exist for a given `(agent_identity, agent_session_id)` pair.

This keeps question ownership on the server while still allowing an Agent platform to scope HITL state per Agent Session.

## MCP Tools

- `hitl_create_question_group`
  Creates a `pending` question group and returns immediately.
- `hitl_wait_question_group`
  Waits for a specific `question_group_id`, or the caller's current pending group if no ID is provided.
- `hitl_get_current_question_group`
  Returns the caller's current pending group.
- `hitl_get_question_group_status`
  Returns lifecycle status for a group by ID.
- `hitl_get_question`
  Returns a question definition by `question_id`.
- `hitl_cancel_question_group`
  Cancels a pending group.

## HTTP API

- `GET /api/v1/healthz`
- `GET /api/v1/readyz`
- `GET /api/v1/metrics`
- `GET /api/v1/question-groups/current`
- `GET /api/v1/question-groups/{question_group_id}`
- `GET /api/v1/questions/{question_id}`
- `PUT /api/v1/question-groups/{question_group_id}/answers/finalize`
- `POST /api/v1/question-groups/{question_group_id}/cancel`
- `POST /api/v1/question-groups/{question_group_id}/expire`

When `HITL_API_KEY` is configured, protected HTTP routes require `x-api-key`. The current-group route also requires `x-agent-session-id`.

## End-To-End Flow

1. Agent calls `hitl_create_question_group`.
2. The server authenticates the caller, reads `x-agent-session-id`, generates `question_group_id`, and stores the group as `pending`.
3. The Agent or backend keeps the returned `question_group_id`.
4. When blocking semantics are needed, the Agent calls `hitl_wait_question_group`.
5. The user answers through your UI.
6. Your backend finalizes answers through HTTP.
7. The server validates answers and transitions the group to `answered`, `cancelled`, or `expired`.
8. Any waiting MCP call is released with the terminal payload.

## Example

Create:

```json
{
  "title": "Release Decision",
  "questions": [
    {
      "question_id": "q_canary",
      "type": "single_choice",
      "title": "Can we start canary deployment?",
      "options": [
        { "value": "yes", "label": "Yes" },
        { "value": "no", "label": "No" }
      ]
    }
  ]
}
```

Wait for current group:

```json
{}
```

Finalize:

```bash
curl -X PUT "http://localhost:3000/api/v1/question-groups/<question_group_id>/answers/finalize" \
  -H "Content-Type: application/json" \
  -H "x-api-key: ${HITL_API_KEY}" \
  -d '{
    "idempotency_key": "idem-release-001",
    "answers": {
      "q_canary": { "value": "yes" }
    },
    "finalized_by": "release-orchestrator"
  }'
```
